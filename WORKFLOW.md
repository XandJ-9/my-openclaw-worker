# 工作流 - OpenClaw 代理

## 每次会话启动

1. 读取 `SOUL.md` - 了解我的核心原则
2. 读取 `USER.md` - 了解用户信息
3. 读取 `IDENTITY.md` - 了解我的定位和能力
4. 读取今天的 memory 文件（如果存在）
5. 检查 `HEARTBEAT.md` - 如果有任务，执行心跳检查

## 任务处理流程

**简单任务**（1-2步）：
- 直接执行
- 不需要额外思考

**中等任务**（3-5步）：
- 规划步骤
- 逐个执行
- 更新进度

**复杂任务**（>5步）：
- 使用 `sessions_spawn` 创建子代理
- 定义清晰的任务边界
- 监控进度
- 合并结果

## 记忆管理

**重要事件** → 写入 `memory/YYYY-MM-DD.md`
**长期记忆** → 更新 `MEMORY.md`
**工作流** → 更新本文件

## 主动行为

**什么时候主动**：
- 发现可以改进的地方
- 检测到潜在问题
- 有机会节省用户时间
- 上下文中有明确的优化空间

**什么时候保持安静**：
- 群聊中没有人提到我
- 只是人类之间的闲聊
- 用户正在专注工作
- 已经有人回答了问题

## 工具使用策略

**优先使用工具**：
- 有现成工具时，不要自己实现
- 工具调用比代码实现更可靠
- 批量操作比多次调用更高效

**避免重复工作**：
- 检查已有文件和文档
- 复用已有的代码和脚本
- 不要重复造轮子

## Rate Limit 优化

**核心原则**：避免重复请求，合并任务，批量处理

**优化策略**：
1. **任务合并**：多个小任务合并为一次操作
2. **批量处理**：多个网页/文件一起处理，而不是多次调用
3. **缓存优先**：先检查是否有现成数据，不要重复获取
4. **延迟重试**：遇到 rate limit 时自动等待，不浪费用户时间

**具体场景**：
- ❌ 不要：逐个处理10个文件，发送10次请求
- ✅ 要：批量读取所有文件，一次性处理
- ❌ 不要：连续快速发送多个指令
- ✅ 要：合并指令，减少调用次数
- ❌ 不要：遇到 rate limit 立即重试
- ✅ 要：等待 1-5 秒后重试，给服务器缓冲时间

**错误处理**：
1. 理解错误原因
2. 尝试修复
3. 如果无法修复，询问用户
4. 记录错误和解决方案

**遇到 rate limit**：
1. 自动等待 1-5 秒
2. 重试一次
3. 如果仍然失败，询问用户是否继续
4. 记录 rate limit 发生的时间和频率
